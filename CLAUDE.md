# CampusLoop 開発ルール

このファイルはこのリポジトリで作業する際の運用ルールをまとめたもの。
Claude はここに書かれたルールに常に従うこと。

## 1. Git運用: 変更ごとにブランチを切ってmainにマージする

コードを変更するときは、mainに直接コミットしない。必ず以下の手順を踏む。

1. 変更内容を意味のある単位に分割する（1つのブランチ/コミットに複数の無関係な
   変更を詰め込まない。例: 「CORS修正」と「セッション認証追加」と「フロント対応」は
   それぞれ別ブランチ・別コミットにする）。
2. 各変更単位ごとに `main` からブランチを切る（命名例: `fix/xxx`, `feature/xxx`,
   `chore/xxx` — このリポジトリの既存ブランチの命名慣習に合わせる）。
3. そのブランチ内で変更を行い、内容が分かるコミットメッセージでコミットする
   （日本語で、何を・なぜ変更したかが分かるように）。
4. `main` に戻り `git merge --no-ff` でマージする（マージコミットメッセージにも
   何をマージしたか分かるように書く）。
5. 複数の変更単位がある場合は、1〜4を単位ごとに繰り返す。

リモート（origin）へのpushは、ユーザーから明示的に指示された場合のみ行う。

## 2. 変更のたびに実行エラーがないか必ず確認する

コードを変更したら、mainにマージする前に必ずエラーチェックを行う。具体的には:

- フロントエンド（`frontend/`）を変更した場合:
  - `cd frontend && npx tsc -b --noEmit` で型エラーがないか確認する
  - `cd frontend && npx eslint .` でLintエラーがないか確認する
  - 両方エラーなしになってから次に進む。エラーが出た場合は原因を修正してから
    コミット・マージする。
- バックエンド（`backend/`）のPHPを変更した場合:
  - 可能であれば `php -l <ファイル>` で構文エラーがないか確認する
    （PHP実行環境がない場合は、コードを読み返して整合性を手動で確認する旨を
    ユーザーに明記する）。
  - ローカルにPostgreSQL/PHPサーバーが立っている場合は、変更したAPIエンドポイント
    を実際に叩いて期待通りのレスポンスが返るか確認する。
- 上記の確認結果（エラーの有無、確認できなかった場合はその理由）を、作業完了の
  報告に含める。「確認していません」で済ませず、確認した/できなかったを明示する。

## プロジェクト概要（参考）

- フロントエンド: React 19 + TypeScript + Vite + Yamada UI（`frontend/`）
- バックエンド: 生PHP + PostgreSQL（`backend/`、フレームワーク不使用）
- 認証: PHPセッションCookie方式（`backend/config/auth.php`）。各APIは
  リクエストの `user_id` を信用せず、必ずセッションから取得した本人IDを使う。
- ローカル起動手順は `LOCAL_DEV.md` を参照。
